# 강인아 포트폴리오 사이트 — 설계 문서 (v4)

작성일: 2026-06-29 · v2: 기술·디자인·범위·접근성·유지보수 리뷰 반영 · v3: UI/UX 심층 리뷰 반영 · v4: 홈 레이아웃을 **"단일 프로그램 창 + 사이드바"** 셸로 확정

> **핵심 원칙 4가지:**
> 1. **사이트 = 하나의 프로그램 창.** 창(타이틀바 + 사이드바)은 항상 떠 있고, 사이드바 네비로 **메인 패널 내용만** 바뀐다. 페이지마다 창을 새로 그리지 않는다 → 일관·단순·"프로그램처럼 둔갑". (§6)
> 2. **모바일 우선** — 사이드바는 폰에서 **하단 탭바**로 변신. 방문자 다수가 폰. 모든 화면에 모바일 동작을 명시한다. (§9)
> 3. **2레이어 목소리** — 사람이 읽는 큰 글씨 = 따뜻한 한국어 입말 / 장식 레이어(`C:\경로`, `guest@`, `//`) = 영어·기호 허용. (§3.4)
> 4. **섀도 = 클릭 가능 신호** — 클릭되는 것만 섀도+hover, 장식(`_ □`·시계)은 흐리게. (§3.3)

## 1. 한 줄 요약

중학생 **강인아**의 개인 포트폴리오 + 블로그 사이트. **"밝은 뉴브루탈리즘 레트로 데스크톱 OS"** 컨셉으로, **사이트 전체가 하나의 레트로 프로그램 창**이고 사이드바(폴더 모양 메뉴: 소개·프로젝트·블로그·연락처)로 둘러본다. 멋있지만 비개발자도 메뉴만 누르면 다 되도록 한다.

## 2. 누구를 위한 것인가

- **주인공:** 강인아 · 대치중학교 2학년 · 학생회 〈가온〉 활동
- **하는 것:** 코딩/개발(게임·웹 등) + 취미·창작(영상·글·그림)
- **연락처:** 이메일, GitHub
- **방문자:** 친구·선생님·가족 등 개발 모르는 사람 + (나중에) 대회·진학에서 작품을 보여줄 사람. 상당수가 **휴대폰**으로 처음 본다.

## 3. 컨셉과 디자인 방향

### 3.1 컨셉 이름
**뉴브루탈리즘(Neo-brutalism) + 레트로 데스크톱 OS** 룩.
- 두꺼운 검은 테두리, 쨍한 플랫 색, 모노/픽셀 글씨 → 뉴브루탈리즘
- 창(window)·타이틀바·닫기버튼·사이드바·폴더 모양 아이콘 → 옛날 컴퓨터 OS 감성
- 사이트 전체를 하나의 프로그램 창으로 만드는 컨셉 = "데스크톱 OS 포트폴리오 / interactive desktop"

### 3.2 접근성 원칙 (가장 중요)
터미널 명령어가 아니라 **"폴더 누르기"**가 기본 입구다. 개발을 몰라도 누구나 쓸 수 있어야 한다. 터미널은 *재미 보너스*이지 *유일한 입구*가 아니다.

**검증 가능한 최소 a11y 기준 (10절 검증과 연결):**
- 모든 폴더·아이콘·버튼은 의미적 요소(`<a>`/`<button>`)로 만든다. 클릭 div 금지. `focus-visible` 포커스 링 + 논리적 Tab 순서.
- 화면의 `C:\강인아\about.exe` 같은 경로 텍스트는 `aria-hidden` 장식이고, **진짜 `<h1>`은 사람이 읽는 제목**("소개", "프로젝트"). 각 창은 `<main aria-label>`.
- `✕`/`← 뒤로` 버튼에 의미 있는 `aria-label`.
- 모든 이미지(썸네일·프로필)에 `alt` 필수.
- "본문 바로가기(Skip to content)" 링크 + `prefers-reduced-motion` 대응(애니메이션 줄이기).

### 3.3 디자인 토큰

**색 (역할별 고정)**
| 역할 | 값 |
|---|---|
| 페이지 배경(창 바깥) + 사이드바 바탕 | `#f3eee1` (크림) + 모눈 그리드 라인 `#e3dcc8`, 간격 24~32px |
| 창/카드/메인 패널 배경 | `#fffdf6` (종이) |
| 잉크(테두리·글자) | `#1a1a1a` |
| 보조 글자 | `#6b6655` (단, 컬러 배경 위에는 쓰지 않음) |
| 사이드바 활성 / UI 포인트 | 민트 `#8fe0bd` (UI 전용, 카테고리 색 아님) |
| 액센트(카테고리) | 스카이 `#7cc0e8` · 코랄 `#f0937a` · 머스터드 `#f5cf52` |

**불변 규칙:** 액센트 배경 위의 글자/아이콘/테두리는 **항상 잉크(`#1a1a1a`)**. **흰색 글자 금지**(컬러 위 흰 글자는 대비 미달 — WCAG AA 실패). 강조가 필요하면 흰 글자 대신 "잉크 글자 + 종이(`#fffdf6`) 안쪽 칩"으로 처리.

**카테고리 → 색 + 표시 라벨 매핑 (상수로 고정, `lib/content.ts`에 둠)**
| frontmatter 키 | 화면 표시 라벨(잠정·말투는 §3.4) | 색 |
|---|---|---|
| 프로젝트 `dev` / 블로그 `dev-log` | "코딩" / "코딩 일기" | 스카이 `#7cc0e8` |
| 프로젝트 `creative` / 블로그 `making` | "창작" / "창작 일기" | 코랄 `#f0937a` |
| 블로그 `thoughts` | "그냥 생각" | 머스터드 `#f5cf52` |

- frontmatter 키(`dev`/`creative`/`dev-log`/`making`/`thoughts`)는 영어로 고정(코드·URL용), **화면 라벨만 한국어 입말**(§3.4).
- 프로젝트와 블로그 칩은 **같은 어휘 계열**을 쓴다("코딩"↔"코딩 일기"). 단독 명사("개발")의 이력서·플랫폼 톤은 피한다.
- 칩에는 **색 + 텍스트 라벨을 항상 함께** 표기(색만으로 구분 금지). 민트·"보라 계열"은 카테고리에서 제외(민트=UI 전용).
- **필터 활성 상태는 색이 아니라 깊이로**: 활성 칩 = 잉크 테두리 굵게 + 안쪽 종이칩 + 체크(또는 더 강한 섀도). 색은 카테고리 표시에만 쓴다.

**테두리·그림자 (단일 토큰으로 고정, 컴포넌트는 이것만 참조)**
- `--border = 2.5px solid #1a1a1a`
- `--shadow = 4px 4px 0 #1a1a1a` (데스크톱) · `--shadow-sm = 2px 2px 0 #1a1a1a` (작은 요소)
- 깊이로 위계: **강조(열린) 카드 `5px` > 일반 카드 `4px` > 칩·사이드바 항목 `2px`** — *색이 아니라 깊이*로 위계 표현
- blur는 항상 `0` (뉴브루탈리즘 핵심)
- 모바일(`<640px`) 오버라이드: `--border = 2px`, `--shadow = 3px 3px 0`
- **섀도 = "클릭 가능" 신호로 예약.** 클릭되는 것(폴더·카드·실동작 버튼)만 섀도+hover 들림을 갖고, **장식 요소(`_ □` 버튼·시계·동작 없는 `시작`)는 섀도·hover·pointer 커서 제거** + 채도 낮춤. 폴더는 정지 화면에서도 카드보다 한 단계 강한 섀도(`5px`)로 "여기가 입구"임을 신호.

**폰트 (2-tier)**
- `--font-display` = **Galmuri11** (한글+영문+모노 픽셀, OFL 라이선스). `next/font/local`로 self-host(정적 export 호환). `font-family: 'Galmuri11', 'Space Mono', monospace`. **적용 범위 = 제목(20px+)·타이틀바·네비 라벨.** 단 12~13px 작은 라벨엔 픽셀 폰트 금지(가독성).
- `--font-body` = 한글 본문 가독성용 시스템 sans(`system-ui`), 추후 Pretendard 검토. **본문·요약·메타데이터·긴 글은 본문 폰트.**
- ⚠️ Space Mono는 한글 글리프가 없으므로 단독 사용 금지. 현재 scaffold가 로드 중인 **Geist는 교체 대상**(12절 1단계).

**다크모드 없음:** 단일 밝은 테마. `html`에 `color-scheme: light only` 지정해 OS 다크모드 강제 반전 차단. `globals.css`의 기존 `prefers-color-scheme: dark` 블록과 기본값(`#ffffff/#171717`)은 위 토큰(크림/종이/잉크)으로 교체(12절 1단계).

### 3.4 카피 / 목소리 가이드 (2레이어 원칙)
이 사이트의 가장 강한 무기는 **"진짜 중2 강인아가 만든 느낌"**이다. 사무적인 명사·영어 클리셰가 이걸 깎아먹으므로, 모든 사용자 글자를 두 레이어로 나눈다. *(아래 문구는 잠정 — 강인아 본인 말투로 바꿔도 됨.)*

- **사람이 읽는 레이어(큰 글씨·인사·소개·라벨·빈상태·에러) = 따뜻한 한국어 입말.**
  - 프로필 인사 `hi! i'm` → **"여기는 강인아"**(또는 "안녕하세요!"). 영어 인사 금지.
  - 프로필 한 줄 = 소속 박제 대신 정체성: **"만들기 좋아하는 중2"**. (`대치중 2학년·학생회 가온`은 `/about` 한 곳에만.)
  - 메인 태그라인은 구체 명사 한 스푼: **"게임이든 영상이든 일단 만들어 보는 게 제일 재밌어요."** `/about` 태그라인은 다른 결로("고치고, 부수고, 다시 만들고").
  - 카테고리 라벨: §3.3 표(코딩 / 창작 / 코딩 일기 / 창작 일기 / 그냥 생각).
  - `about` 섹션 헤더: `// 관심사` 같은 코드 주석 톤 대신 평이하게("요즘 빠진 것", "요즘 쓰는 도구"). 사이트 전역 섹션 헤더는 **한 가지 규격**으로 통일.
- **장식 레이어(타이틀바 경로 `C:\강인아\about.exe`, `guest@kanglena:~$`, 터미널 출력) = 영어·기호 OK.** 단 §3.2대로 `aria-hidden`(스크린리더가 안 읽음).
- **터미널 메시지도 한국어 입말**: 알 수 없는 명령 → `음, xxx는 모르는 명령이에요. help를 쳐 보세요`. `whoami` → `강인아 · 중2 · 만드는 사람`.

## 4. 기술 스택 / 아키텍처

- **이미 설치된 것 그대로:** Next.js 16.2.9 (App Router) · React 19 · Tailwind v4 · TypeScript
- **콘텐츠:** 마크다운 파일. `.md` 하나 = 글/작업 하나. 빌드 시 읽어서 정적 페이지로 생성
- **배포:** GitHub Pages 무료. Next.js **정적 export**(`output: 'export'`)로 `out/` 생성 → GitHub Actions로 자동 배포
- **상태:** 백엔드·DB·로그인 없음. 100% 정적 사이트

> ⚠️ **구현 전 필독:** 저장소 `AGENTS.md`에 따르면 이 Next.js는 학습 데이터와 다른 breaking change가 있을 수 있다. 코드 작성 전 `node_modules/next/dist/docs/`(특히 `01-app/02-guides/upgrading/version-16.md`, 정적 export, 메타데이터 문서)를 읽고 deprecation을 확인한다.

### 4.1 정적 export / 배포
**`next.config.ts` (현재 비어 있음 → 반드시 채움):**
- `output: 'export'`
- `images: { unoptimized: true }` (정적 export는 이미지 최적화 불가). `images.domains`는 Next 16에서 deprecated이니 쓰지 않음
- `trailingSlash: true` (GitHub Pages 새로고침 404·자원 경로 어긋남 방지 + 로컬 dev와 배포 결과 차이 줄임)
- `cacheComponents`(use cache/PPR)는 **사용 안 함**(기본 끔 유지). 켜면 콘텐츠 0개일 때 `generateStaticParams` 빈 배열이 빌드를 깨뜨려 "빈 껍데기 먼저 배포"가 실패함

**배포 (GitHub Actions):** `.github/workflows/`에 워크플로 구성 — `configure-pages` → build → `touch out/.nojekyll` → `upload-pages-artifact` → `deploy-pages`. 워크플로 `permissions`에 `pages: write`, `id-token: write`. `upload-pages-artifact` 방식이라 `out/`이 `.gitignore`여도 무방.
- ⚠️ **`.nojekyll` 없으면** Jekyll이 `_next/` 폴더를 통째로 무시 → 스타일·인터랙션 다 깨진 흰 화면.
- 저장소명이 `kanglena.github.io` → **유저 사이트로 가정**(루트 `https://kanglena.github.io/`), `basePath` 불필요. *구현 시 저장소 종류 확인, 프로젝트 저장소면 `basePath`/`assetPrefix` 설정.*

**최초 1회 수동 체크리스트 (README에 기록):**
1. GitHub → Settings → Pages → Source = **"GitHub Actions"** (브랜치 배포 아님)
2. 흰 화면이 뜨면 → 브라우저 개발자도구 콘솔에서 `_next` 404 확인 → `.nojekyll` 누락 점검

### 4.2 콘텐츠 파이프라인
- `content/projects/*.md`, `content/posts/*.md` — **폴더는 평평하게**(하위 카테고리 폴더 금지, category는 frontmatter 필드로). 파싱 복잡도만 늘어남(YAGNI)
- `gray-matter`로 frontmatter 파싱 + **가벼운 단일 마크다운 변환기**(`marked` 또는 `markdown-it`)로 본문 → HTML. (remark/rehype는 패키지 5~8개로 늘고 초보자 디버깅이 어려움. 코드 하이라이트·각주가 당장 불필요하니 가벼운 쪽이 기본값.)
- `lib/content.ts`는 **server-only**(`'use client'`에서 import 금지). 조회 함수는 `collection` 인자 하나로 합쳐 복제 줄임: `getContent('posts')`, `getContent('projects')`, `getOne('posts', slug)` (날짜 내림차순, `draft: true` 제외)
- **친절한 검증 함수(빌드 실패 + 한글 안내):** 필수 필드(`title`, `date`, `category`) 누락이나 category 오타 시 "파일명 + 무엇이 잘못 + 어떻게 고치는지"를 한국어로 출력. 예: `content/posts/x.md: date가 없어요. 맨 위에 "date: 2026-06-29" 형식으로 추가하세요`. category는 허용값(`projects: dev|creative`, `posts: dev-log|making|thoughts`)과 대조. **zod 등 스키마 라이브러리는 쓰지 않음**(과함).
- **slug 규약:** `.md` 파일명(=slug)은 **영문 소문자 + 하이픈만**. 한글/공백 파일명은 GitHub Pages에서 URL 인코딩이 어긋나 404 위험. `lib/content.ts`에서 검증.

### 4.3 데이터 모델 (frontmatter)
**프로젝트 (`content/projects/*.md`)**
```yaml
title: 미니 타자 게임
summary: 파이썬으로 만든 첫 게임
category: dev          # dev | creative
tags: [Python, Pygame]
date: 2026-06-20       # 반드시 YYYY-MM-DD
tools: [Python, Pygame]
links: { github: "...", demo: "...", video: "..." }   # 있는 것만
thumbnail: /images/typing-game.png                     # 선택(맨 앞 슬래시 포함)
draft: false
```
**블로그 (`content/posts/*.md`)**
```yaml
title: 처음으로 게임을 만들어봤다
date: 2026-06-20
category: dev-log      # dev-log | making | thoughts
tags: [회고, Python]
summary: 만들면서 부딪힌 것들
thumbnail: /images/...   # 선택
draft: false
```
**이미지 규칙:** 모든 이미지는 `public/images/`에 두고 어디서든 `/images/파일이름.png`(맨 앞 슬래시 포함)로 참조(본문 마크다운도 동일). `unoptimized: true`라 자동 압축이 없으니 **올리기 전 가로 1200px 이하 + 1MB 넘으면 squoosh.app 등으로 압축**. 파일명은 글 이름을 앞에 붙이는 가벼운 컨벤션(`typing-game-1.png`).

### 4.4 Next 16 라우팅 주의 (구현 함정)
- page/layout의 `params`·`searchParams`는 **Promise**다(동기 접근 제거됨). 모든 동적 페이지는 `export default async function Page(props){ const { slug } = await props.params }` 형태. `npx next typegen`으로 `PageProps` 헬퍼 타입 생성.
- 정적 export는 `dynamicParams` 금지 → **모든 slug 경로를 `generateStaticParams`로 전부 생성.**
- 코드 작성 전 `node_modules/next/dist/docs/01-app/02-guides/upgrading/version-16.md` 필독.

## 5. 페이지 / 정보 구조

**전체 셸 = 하나의 프로그램 창** (모든 라우트가 공유):
- **타이틀바**: `C:\강인아 — portfolio` + `_ □ ✕`(장식, §6.2) + (선택) GitHub·메일 즉시동작 아이콘
- **사이드바 네비**(데스크톱 좌측 / 모바일 하단 탭바): `홈 · 소개 · 프로젝트 · 블로그 · 연락처` + (보너스) `터미널`. 현재 위치는 활성 표시(§3.3 깊이).
- **메인 패널**: 라우트별 내용만 여기서 바뀜.

| 경로 | 메인 패널 내용 |
|---|---|
| `/` | **홈(대시보드)** — 사이트 정의 한 줄 + 프로필 요약(정체성 한 줄 + "소개 더보기→/about") + 최근 작업·글 미리보기 |
| `/about` | 프로필, 한 줄 소개, 긴 소개, 관심사 태그, 요즘 쓰는 도구 태그, 연락(이메일·GitHub) |
| `/projects` | 필터 칩(전체·코딩·창작) + 카드 그리드 |
| `/projects/[slug]` | 제목·요약·분류·태그·기간·도구·링크 + 본문(왜·어떻게·배운 점) + 이미지/영상 |
| `/blog` | 카테고리 칩 + 글 목록 |
| `/blog/[slug]` | 제목·날짜·태그 + 본문(마크다운 렌더) |
| `/contact` | 이메일·GitHub (§6.6) |
| `/terminal` | 보너스, §6.4 |

**홈(`/`) 메인 패널 구성·위계:** 위→아래 크기·여백으로 위계.
1. **사이트 정의 한 줄**(본문 폰트) — 예: "강인아의 작업과 글을 모아둔 공간이에요." (5초 안에 *무엇인지* 전달.)
2. **프로필 요약** — 한국어 인사("여기는 강인아") + 정체성 한 줄("만들기 좋아하는 중2") + "소개 더보기 → /about". 전체 소개는 `/about`에서만.
3. **최근 작업·글 미리보기** — 카테고리색 썸네일 헤더가 있는 **예쁜 카드 그리드**(밋밋한 목록 X). 각 카드는 **상세로 직행하는 의미적 `<a>`**(전체가 클릭 타깃, 모바일 44px+, hover 들림). "더보기 → /projects·/blog"는 별도 링크. **콘텐츠 0개면 섹션 숨김**(미완성 인상 방지).

> 사이드바 네비 항목은 **폴더/앱 아이콘 스타일**(컬러 사각형 + 아이콘 + 라벨)로 OS 감성 유지. 길찾기는 항상 보이는 사이드바(모바일=탭바)가 담당하므로 어디서든 길 잃음이 없다.

## 6. 인터랙션 모델

### 6.1 기본 (단일 프로그램 창 셸)
- 사이트 전체가 **하나의 지속 창**(타이틀바 + 사이드바 + 메인 패널). **사이드바 네비를 클릭 → 메인 패널 내용만 교체**(라우트 이동). 창·타이틀바·사이드바는 그대로 유지.
- 실제 라우팅은 평범한 페이지 이동(정적 export). 셸(타이틀바·사이드바)은 `app/layout.tsx`(또는 공통 레이아웃)에 한 번만 두어 모든 페이지가 공유.
- **싱글클릭으로 동작.** 현재 위치는 사이드바에서 활성 표시(§3.3 깊이) → 길 잃음 없음.
- 드래그·다중창 매니저는 만들지 않는다(YAGNI, 모바일·접근성 우선).

### 6.2 뒤로/닫기
- **길찾기는 사이드바(모바일=하단 탭바)가 담당** → 별도 "✕=상위 화면" 규칙이 불필요. 상세→목록 복귀는 사이드바의 해당 항목 또는 본문 상단의 `← 목록으로` 링크로.
- 타이틀바의 `✕`는 **장식**(§3.3 섀도 규칙대로 흐리게, `aria-hidden`). `_ □`도 동일. (`□`를 실제 전체화면 토글로 구현하는 건 선택.)
- **필터 보존(중요):** 목록↔상세 왕복에서 필터가 풀리지 않게, 카드→상세 링크에 현재 필터 쿼리를 싣고 `← 목록으로`도 그 쿼리로 돌아간다(예: `/projects?cat=dev`). `router.back()` 의존 금지(공유 링크로 상세에 직접 들어온 방문자에게 깨짐). 빌드 후 실제 클릭으로 검증.

### 6.3 사이드바/탭바 어포던스 (클릭 가능 신호)
뉴브루탈리즘은 모든 요소가 테두리+그림자라 "눌러진다"는 차별화가 약하다. 보강:
- 네비 항목 hover/focus 시 그림자가 커지거나 살짝 들림 + `cursor:pointer`, 활성 항목은 채워진 컬러칩으로 구분.
- 데스크톱 사이드바 항목에 한 줄 설명 툴팁/부제("내가 만든 것들") 가능. 모바일 탭바는 아이콘 + 짧은 라벨.

### 6.3-b 본문 상단 프롬프트는 제목의 들러리로
`$ ls ./projects` 같은 모노 프롬프트를 쓸 경우 **작은 키커**(보조색 `#6b6655`, 12~13px)로 강등하고, 진짜 `<h1>`("프로젝트")이 그 자리 최대 글자가 되게 한다. 중복이면 빼도 됨. **모바일(`<640px`)에선 `display:none`**(픽셀 폰트가 좁은 폭에서 줄바꿈돼 깨져 보임).

### 6.4 터미널 (보너스)
`'use client'`. 명령별 동작을 표로 고정:
| 명령 | 동작 |
|---|---|
| `about` / `projects` / `blog` / `contact` | 해당 라우트로 이동(`router.push`) |
| `help` / `whoami` / `clear` | 메인 패널의 터미널 영역에 텍스트 출력 |
| 그 외 | `음, xxx는 모르는 명령이에요. help를 쳐 보세요` 친절 피드백 |
- 진입 시 환영 메시지 + **클릭 가능한 명령어 칩**(타이핑 못 해도 사용 가능). `exit`로 홈 복귀.
- 가상 파일시스템(`ls`/`cd`/`cat` 흉내) 없음.
- **본동선의 느린 복제가 아니라 "재미 보너스"로 차별화:** `whoami`·`help`·작은 이스터에그 같은 *표현/재미* 출력 중심. 모든 메시지는 한국어 입말(§3.4). 길찾기는 폴더가 담당하고 터미널은 "장난감"이라는 정체성을 유지.

### 6.5 클라이언트 컴포넌트 정적 export 안전 패턴
- 터미널·`ContentFilter` 등 `'use client'` 컴포넌트는 `window`/`localStorage`를 모듈 최상위·렌더 본문에서 직접 접근하면 빌드타임 prerender에서 깨짐 → **`useEffect` 안에서만** 접근.
- 타이틀바/셸의 시계(있으면)는 초기값 `null` + `useEffect`로 설정해 hydration mismatch 방지.

### 6.6 연락 동선 (입구 일원화)
"연락"의 입구가 여러 곳(사이드바 연락처 · `/about` 하단 · 타이틀바 메일)이라 동작이 제각각이면 혼란스럽다. **동작을 고정한다:**
- 타이틀바·푸터의 메일·GitHub = **즉시 동작**(`mailto:`/새 탭 GitHub), `aria-label`로 예고("이메일 보내기").
- `/contact`는 단순 중복을 피해 *차별점*을 준다("연락 시 이런 내용 적어주면 좋아요").
- **선택(초보 친화 단순화):** 사이드바 네비를 3개(소개·프로젝트·블로그)로 줄이고 연락을 타이틀바 + `/about`로 일원화해도 됨. → §16 "열린 결정"에서 택1.

## 10. 빈 상태 / 에러 처리

- **빈 상태는 상황별로 분기**(같은 문구로 때우지 않기):
  - 컬렉션 자체가 빔 → "첫 작품 준비 중이에요, 곧 올라와요."
  - 필터 0건 → "'창작'으로 분류된 건 아직 없어요. '전체'를 눌러 보세요." (해결 동작을 제시)
  - 홈 '최근' 섹션이 0개 → 섹션 자체를 숨김(빈 껍데기 노출 금지, §5).
- **404 (`not-found.tsx`):** 정적 export가 `out/404.html`로 내보내 GitHub Pages 커스텀 404로 동작. **동적 서버 함수(`headers()`/`cookies()`) 미사용 순수 정적**으로 작성(`usePathname` 같은 클라이언트 훅은 별도 컴포넌트로 분리). `experimental global-not-found`는 사용 안 함. **막다른 골목 → 분기점:** 큰 "홈으로" 버튼 + 프로젝트·블로그·소개 바로가기 2~3개를 함께 배치(공유 링크로 떨어진 사람이 헤매지 않게). 사이드바/탭바는 셸에 그대로 있어 404에서도 길찾기가 살아 있음. 캐릭터 한 스푼의 평이한 한국어("주소가 틀렸거나, 제가 아직 안 만든 페이지예요").
- 깨진/누락 frontmatter: 4.2절 검증 함수가 빌드 실패로 알려줌(조용히 넘어가지 않기).

## 11. 테스트 / 검증 전략

- **핵심 게이트 = `next build`(정적 export) 성공.** GitHub Actions에서 자동 실행 → 사실상 통합 테스트. **콘텐츠 0개 상태에서도 빌드 성공**해야 함(첫 배포).
- 빌드 산출물 확인: `out/`에 `404.html`, `sitemap.xml`, `robots.txt`, OG 메타태그, `.nojekyll` 존재.
- `lib/content.ts` 단위 테스트는 **frontmatter 검증 함수에만 좁게**(vitest 1개). 커버리지 확대에 시간 쓰지 않음.
- CI에 `eslint`를 빌드와 **별개 단계**로 추가(Next 16은 build가 lint를 돌리지 않음).
- 접근성: 배포 후 **Lighthouse 접근성 점검 또는 axe 수동 점검 1회**.
- 배포 후 `kanglena.github.io` 실제 확인(모바일 포함).

## 13. 단계별 구현 순서

1. **뼈대(셸)** — `next.config.ts` 정적 export 설정 + `globals.css`를 디자인 토큰으로 교체(다크 블록 제거) + **Geist→Galmuri11/본문폰트 교체** + **`AppShell`(타이틀바+사이드바/탭바)·`SideNav`·`NavItem`** + 커스텀 favicon. 이 셸이 모든 페이지를 감싼다.
2. **홈(`/` 메인 패널)** — 사이트 정의 한 줄 + 프로필 요약 + 최근 미리보기(예쁜 카드)
3. **소개** 페이지
4. **콘텐츠 파이프라인 + 프로젝트** — `lib/content.ts`(파싱·검증) + 목록/필터/상세 + **예시 콘텐츠(프로젝트 2개)**
5. **블로그** — 목록/상세(4번 패턴 재사용) + **예시 글 2개**
6. **SEO/메타데이터 + 연락처 + 404 + 반응형 마무리**
7. **GitHub Pages 자동 배포**(워크플로 + `.nojekyll` + 최초 1회 수동 설정)
8. **(보너스) 터미널** — 1~7 완료·배포 후

**배포 전 필수 게이트:** 4·5단계에서 "진짜처럼 보이는" 더미 콘텐츠 **프로젝트 2 + 글 2**를 반드시 커밋한다(화면이 살아 있게 + 정답 템플릿 + 픽스처 역할). 콘텐츠 0개로 첫 배포하면 홈 '최근' 섹션이 빈 껍데기가 돼 친구·가족·심사자에게 '미완성' 첫인상을 준다.

## 14. 콘텐츠 작성 가이드 (산출물: `content/README.md`)

코드보다 먼저/같이 만든다(6개월 뒤 본인이 잊는 게 최대 위험). **한국어**로:
- **새 글 추가 3단계:** 폴더에서 `_template.md` 복사 → 내용 수정 → 저장
- 각 폴더(`posts`, `projects`)에 복사용 **`_template.md`** 실제 커밋
- frontmatter "되는 것/안 되는 것" 표 — 특히 `date`는 반드시 `YYYY-MM-DD`, 제목에 콜론(`:`)이 들어가면 따옴표로 감싸기, `tags`는 대괄호 배열
- 파일명(=slug)은 **영문 소문자-하이픈만**(URL이 됨)
- 이미지: `public/images/`에 두고 `/images/이름.png`로 참조, 올리기 전 압축

**`README.md` 맨 위(큰 글씨) 로컬 미리보기 2줄:**
- 글 쓸 때: `npm run dev` (저장 시 즉시 새로고침)
- 올리기 전 최종 확인: `npm run build && npx serve out` (산출물은 `.next`가 아니라 `out/`)
